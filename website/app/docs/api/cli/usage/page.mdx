# CLI usage

After reading this page you can run any `docs` sub-command with the right flags and understand what each option does, what values it accepts, and when to reach for it in CI.

## How it works

The `@farming-labs/docs` CLI is invoked via your package manager's exec helper or directly with `npx`:

```bash
npx @farming-labs/docs@latest <command> [options]
pnpm exec docs <command> [options]
```

All commands read your `docs.config.ts` (or `docs.config.tsx`) from the project root by default. Pass `--config <path>` to override.

---

## `docs doctor` flags

`docs doctor` audits your docs site and exits with a non-zero code when checks fail. It has two scoring modes — agent-readiness (default) and reader-facing quality — and several flags that control what gets checked and how failures are reported.

### Mode flags

| Flag | Description |
|---|---|
| `--agent` | Score agent-readiness for the current docs app. Default when no mode flag is given. |
| `--site` | Score reader-facing docs quality (navigation, descriptions, structure, trust signals, feedback). Alias: `--human`. |

You can also use the subcommand form:

```bash
pnpm exec docs doctor agent
pnpm exec docs doctor site
```

### Output flags

| Flag | Description |
|---|---|
| `--json` | Print the full report as JSON instead of the human-readable summary. |
| `--url <url>` | Run checks against a live deployed URL instead of the local filesystem. |

### CI and failure flags

| Flag | Description |
|---|---|
| `--strict` | Exit with failure when any check warns **or** fails. |
| `--fail-on warn\|fail` | Choose the threshold at which the command exits non-zero. `warn` exits on any warning or failure. `fail` exits only when a check hard-fails. |

`--fail-on` accepts exactly two values:

- **`warn`** — the command exits non-zero if any check produces a warning or a failure.
- **`fail`** — the command exits non-zero only when a check hard-fails (warnings are tolerated).

Passing any other value, an empty string, or omitting the value after the flag throws an error:

```
Missing value for --fail-on.
Invalid value for --fail-on. Expected warn or fail.
```

<Callout type="tip" title="--strict vs --fail-on">
`--strict` is equivalent to `--fail-on warn`. Use `--fail-on fail` when you want CI to pass through warnings but still catch hard failures.
</Callout>

### Filter flags

| Flag | Description |
|---|---|
| `--only agent` | Run only agent-readiness checks. |
| `--only site` | Run only reader-facing checks. |

### Examples

```bash
# Default agent audit
pnpm exec docs doctor

# Reader-facing audit
pnpm exec docs doctor --site

# Agent audit, machine-readable output
pnpm exec docs doctor --agent --json

# Fail CI on warnings too
pnpm exec docs doctor --agent --strict

# Fail CI only on hard failures, tolerate warnings
pnpm exec docs doctor --agent --fail-on fail

# Check a live deployed site
pnpm exec docs doctor --agent --url https://docs.example.com
```
