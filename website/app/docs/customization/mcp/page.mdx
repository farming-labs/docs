# MCP Server

<Agent>
MCP is enabled and public by default: `/mcp` and `/.well-known/mcp` forward to the canonical
`/api/docs/mcp` handler. Add `mcp.security.authenticate` only for private docs; return a principal to
continue or `null` for `401`. Add `mcp.security.protectedResource.authorizationServers` and scopes
only when OAuth-aware clients need RFC 9728 discovery.

Run `pnpm exec docs doctor --agent`, then initialize a client and call `list_docs` followed by
`read_page` for a known route. A `404` can mean MCP is disabled, static export omitted the server
route, or the canonical handler and public forwarder are missing or disagree; use doctor output to
distinguish them. Repeated `401` responses mean the callback is not returning a valid principal;
`403` with `insufficient_scope` means the principal lacks a configured required scope. Local-only
clients can avoid HTTP routing with `pnpm exec docs mcp`.
</Agent>

`@farming-labs/docs` can expose your docs as a built-in MCP server, so AI clients can search the docs, read pages, list sections, inspect code examples, and understand config options without scraping HTML.

The current built-in surface includes:

- `list_docs`
- `list_pages`
- `get_navigation`
- `search_docs`
- `read_page`
- `get_code_examples`
- `get_config_schema`
- `get_context`

It also exposes resources for:

- `docs://navigation`
- one `docs://…` resource per page

## HTTP security and opt-in authentication

The HTTP MCP endpoint is **public by default**. Authentication only becomes required when you add
`mcp.security.authenticate`. This lets public documentation stay zero-config while private docs can
reuse any authentication provider that can inspect a Web `Request`.

Adding `authenticate` by itself keeps the existing callback-only contract. Add
`security.protectedResource` alongside it when OAuth-aware MCP clients should discover an
authorization server, requested scopes, and RFC 9728 protected-resource metadata.

Even for public endpoints, the HTTP transport validates a supplied `Origin` header against the
request origin and limits POST bodies to 1 MiB by default. Origin-less requests remain supported
for non-browser MCP clients. The body is buffered through that limit before an Origin or
authentication callback runs, so callbacks never receive an oversized POST. These HTTP controls do
not affect the local stdio command.

```ts title="docs.config.ts"
export default defineDocs({
  mcp: {
    security: {
      async authenticate({ request }) {
        const session = await getSession(request.headers);

        if (!session) return null;

        return {
          id: session.user.id,
          scopes: ["docs:read"],
          claims: {
            organizationId: session.user.organizationId,
          },
        };
      },
    },
  },
});
```

The callback result controls the request:

- Return `{ id, scopes?, claims? }` to continue as an authenticated principal.
- Return `null` to receive `401 Unauthorized`. With `protectedResource`, the response also includes
  a Bearer `WWW-Authenticate` challenge that points to the resource metadata.
- Return a Web `Response` to control the rejection, redirect, or challenge yourself. The framework
  preserves its status, body, and headers instead of adding an OAuth challenge.
- Throw to receive a sanitized `500` response; provider error details are not exposed.

### OAuth protected-resource discovery

Configure both `authenticate` and `protectedResource` to opt into RFC 9728 discovery and
framework-generated Bearer challenges:

```ts title="docs.config.ts"
export default defineDocs({
  mcp: {
    security: {
      protectedResource: {
        authorizationServers: ["https://auth.example.com"],
        scopesSupported: ["docs:read", "docs:search"],
        requiredScopes: ["docs:read"],
        resourceName: "Private product docs",
        resourceDocumentation: "https://docs.example.com/auth/mcp",
      },
      async authenticate({ request, resource }) {
        const token = await verifyAccessToken(request);

        if (!token) return null;
        if (!token.audiences.includes(resource)) return null;

        return {
          id: token.subject,
          scopes: token.scopes,
        };
      },
    },
  },
});
```

`protectedResource` makes the docs endpoint an OAuth resource server; it does not issue or validate
tokens. The callback must read the Bearer token, validate its signature, issuer, expiry, and the
exact `resource` audience supplied in the callback context, and return only scopes verified from
that token. The `request` retains the real incoming URL, including its query string. Never pass an
upstream token through to another service. The configured authorization server must publish RFC
8414 or OpenID Connect discovery and support a client-registration approach your MCP clients can
use.

`authorizationServers` is required and must contain at least one HTTPS authorization-server issuer
URL without a query or fragment; loopback HTTP is accepted for development. `scopesSupported` is
advertised to clients. `requiredScopes` is enforced against the
`scopes` returned by `authenticate` for every HTTP MCP request.

When this configuration is active:

- missing or invalid credentials that make `authenticate` return `null` receive `401` with a
  Bearer challenge containing `resource_metadata` and the required `scope`
- an authenticated principal missing any `requiredScopes` receives `403` with
  `error="insufficient_scope"`
- a `Response` returned by `authenticate` remains authoritative, so add `WWW-Authenticate` to that
  response yourself when a custom rejection should participate in OAuth discovery
- the stdio transport is unchanged

The framework publishes a public metadata document for each MCP HTTP alias:

```txt title="RFC 9728 protected-resource metadata"
/.well-known/oauth-protected-resource/mcp
/.well-known/oauth-protected-resource/.well-known/mcp
/.well-known/oauth-protected-resource/api/docs/mcp
```

If `mcp.route` is customized, the last path uses that configured route instead. Each document's
`resource` value identifies the corresponding public MCP URL, and its
`authorization_servers`, optional `scopes_supported`, resource name, and documentation URL come
from `protectedResource`. Metadata stays public so a client can discover how to authenticate before
it has a token. `GET` and `HEAD` are supported, and browser discovery receives public CORS headers.

Each alias is a distinct OAuth resource identifier. Configure token audience validation for the
exact endpoint your clients use; prefer `/mcp` as the shared public endpoint unless clients require
the canonical or well-known alias.

`protectedResource` is inactive without `authenticate`: the MCP endpoint remains public and the
metadata routes return `404`. This prevents discovery metadata from claiming that a public endpoint
is protected.

Generated TanStack Start, SvelteKit, and Astro forwarders pass `config.mcp` to
`isDocsMcpRequest(url, config.mcp)`. Update that call when enabling protected auth in a project
generated by an older release. The legacy one-argument form continues to match MCP endpoints but
deliberately does not reserve OAuth metadata paths in public projects.

<Callout type="warning" title="Use HTTPS for hosted OAuth">
  Use absolute `https://` authorization-server and resource-documentation URLs in production, and
  serve the hosted MCP endpoint over HTTPS. Plain HTTP should be limited to loopback development.
</Callout>

<Callout type="warning" title="Next.js basePath">
  Protected MCP cannot use Next.js `basePath`: RFC 9728 inserts its well-known path at the origin
  root, while Next.js only supports disabling `basePath` for external rewrites. Host the protected
  docs app at the origin root, or publish the MCP proxy and protected-resource metadata at the edge.
  For an inline MCP object, `withDocs()` detects this combination. If the MCP configuration is
  imported, spread, or shorthand, pass the live docs config as the second argument so protected
  rewrites and the `basePath` check use the resolved object:

  ```ts title="next.config.ts"
  import { withDocs } from "@farming-labs/next/config";
  import docsConfig from "./docs.config";

  export default withDocs({}, docsConfig);
  ```
</Callout>

Custom low-level `DocsMcpSource` implementations receive that principal in the optional second
argument to `getPages(locale, context)` and `getNavigation(locale, context)`. They can use scopes or
claims to filter private content. The framework does not serialize the principal into MCP tool
results or telemetry.

Configure browser origins and the request limit independently:

```ts title="docs.config.ts"
export default defineDocs({
  mcp: {
    security: {
      allowedOrigins: [
        "https://docs.example.com",
        "https://internal-tools.example.com",
      ],
      maxBodyBytes: 512 * 1024,
    },
  },
});
```

`allowedOrigins` also accepts `"same-origin"` (the default) or an async policy callback:

```ts title="docs.config.ts"
export default defineDocs({
  mcp: {
    security: {
      allowedOrigins: async ({ origin, request }) => {
        const hostname = new URL(origin).hostname;
        return (
          origin === new URL(request.url).origin ||
          hostname === "example.com" ||
          hostname.endsWith(".example.com")
        );
      },
    },
  },
});
```

### Browser CORS

Cross-origin browser access is opt-in through `allowedOrigins`. When an Origin is accepted, the
framework answers `OPTIONS` preflight without running `authenticate` and emits that exact Origin in
`Access-Control-Allow-Origin`. It never combines wildcard Origin responses with credentials.

The standard MCP request headers, including `Authorization`, `Content-Type`,
`MCP-Protocol-Version`, and `MCP-Session-Id`, are allowed by default. Add application-specific auth
headers explicitly:

```ts title="docs.config.ts"
export default defineDocs({
  mcp: {
    security: {
      allowedOrigins: ["https://internal-tools.example.com"],
      cors: {
        allowedHeaders: ["X-API-Key"],
        exposedHeaders: ["X-Docs-Version"],
        allowCredentials: true,
        maxAgeSeconds: 600,
      },
    },
  },
});
```

`allowCredentials` is `false` by default. Enable it only for an explicitly allowed browser Origin
when the auth provider needs cookies or another browser credential. Set `cors: false` to keep Origin
validation but suppress browser CORS headers.

Generated routes and public forwarders include `OPTIONS`. On a manually maintained MCP route,
export or forward `MCP.OPTIONS` alongside `GET`, `POST`, and `DELETE`.

<Callout type="warning" title="Origin is not authentication">
  Origin validation protects the browser-facing transport; it does not identify a user or agent.
  Use `authenticate` when the documentation itself needs access control. Headers such as
  `Signature-Agent` are content-negotiation hints and must not be treated as proof of identity.
</Callout>

## Default HTTP Routes

`/mcp` is the short public MCP endpoint in Next.js, and `/.well-known/mcp` is the discovery-friendly
public MCP endpoint. Both rewrite to the canonical framework route at `/api/docs/mcp`, so the MCP
handler still lives in one place.

<Callout type="info" title="Minimal route behavior">
  **Next.js** auto-generates the default `/api/docs/mcp` route when you use `withDocs()`, and also
  adds public `/mcp` and `/.well-known/mcp` rewrites unless you explicitly disable MCP.

  **TanStack Start**, **SvelteKit**, **Astro**, and **Nuxt** use one public forwarder each so
  `/api/docs/mcp`, `/mcp`, and `/.well-known/mcp` share the same built-in handler without creating
  one route file per alias.
</Callout>

With `withDocs()`, no extra route file is needed for the default path.

### TanStack Start

```ts title="src/routes/$.ts"
import { createFileRoute } from "@tanstack/react-router";
import {
  isDocsMcpRequest,
  isDocsPublicGetRequest,
  isDocsStandardsDiscoveryRequest,
} from "@farming-labs/docs";
import { docsServer } from "@/lib/docs.server";
import docsConfig from "../../docs.config";

const docsEntry = docsConfig.entry ?? "docs";

async function handlePublicDocsRequest(request: Request) {
  const url = new URL(request.url);
  const method = request.method.toUpperCase();

  if (isDocsMcpRequest(url, docsConfig.mcp)) {
    if (method === "OPTIONS") return docsServer.MCP.OPTIONS({ request });
    if (method === "POST") return docsServer.MCP.POST({ request });
    if (method === "DELETE") return docsServer.MCP.DELETE({ request });
    if (method === "GET" || method === "HEAD") return docsServer.MCP.GET({ request });
    return new Response("Method Not Allowed", {
      status: 405,
      headers: { Allow: "GET, HEAD, POST, DELETE, OPTIONS" },
    });
  }

  if (
    isDocsStandardsDiscoveryRequest(url, {
      apiRoute: docsConfig.cloud?.apiRoute,
    })
  ) {
    if (method === "HEAD") return docsServer.HEAD({ request });
    if (method === "POST") return docsServer.POST({ request });
    return docsServer.GET({ request });
  }

  if (
    (method === "GET" || method === "HEAD") &&
    isDocsPublicGetRequest(docsEntry, url, request, {
      apiRoute: docsConfig.cloud?.apiRoute,
      sitemap: docsConfig.sitemap,
      llms: docsConfig.llmsTxt,
      robots: docsConfig.robots,
    })
  ) {
    return method === "HEAD" ? docsServer.HEAD({ request }) : docsServer.GET({ request });
  }

  return new Response("Not Found", { status: 404 });
}

export const Route = createFileRoute("/$")({
  server: {
    handlers: {
      GET: async ({ request }) => handlePublicDocsRequest(request),
      HEAD: async ({ request }) => handlePublicDocsRequest(request),
      POST: async ({ request }) => handlePublicDocsRequest(request),
      DELETE: async ({ request }) => handlePublicDocsRequest(request),
      OPTIONS: async ({ request }) => handlePublicDocsRequest(request),
      ANY: async ({ request }) => handlePublicDocsRequest(request),
    },
  },
});
```

### SvelteKit

```ts title="src/hooks.server.ts"
import {
  isDocsLlmsTxtPublicRequest,
  isDocsMcpRequest,
  isDocsPublicGetRequest,
  isDocsStandardsDiscoveryRequest,
} from "@farming-labs/docs";
import config from "$lib/docs.config";
import { GET, HEAD, POST, MCP } from "$lib/docs.server";

const docsEntry = config.entry ?? "docs";

export async function handle({ event, resolve }) {
  const method = event.request.method.toUpperCase();

  if (isDocsMcpRequest(event.url, config.mcp)) {
    if (method === "OPTIONS") return MCP.OPTIONS({ request: event.request });
    if (method === "POST") return MCP.POST({ request: event.request });
    if (method === "DELETE") return MCP.DELETE({ request: event.request });
    if (method === "GET" || method === "HEAD") return MCP.GET({ request: event.request });
    return new Response("Method Not Allowed", {
      status: 405,
      headers: { Allow: "GET, HEAD, POST, DELETE, OPTIONS" },
    });
  }

  if (
    isDocsStandardsDiscoveryRequest(event.url, {
      apiRoute: config.cloud?.apiRoute,
    })
  ) {
    if (method === "HEAD") return HEAD({ url: event.url, request: event.request });
    if (method === "POST") return POST({ url: event.url, request: event.request });
    return GET({ url: event.url, request: event.request });
  }

  if (
    (method === "GET" || method === "HEAD") &&
    isDocsLlmsTxtPublicRequest(event.url, config.llmsTxt, docsEntry, {
      apiRoute: config.cloud?.apiRoute,
    })
  ) {
    const nativeResponse = await resolve(event);
    if (nativeResponse.status !== 404) return nativeResponse;
  }

  if (
    (method === "GET" || method === "HEAD") &&
    isDocsPublicGetRequest(docsEntry, event.url, event.request, {
      apiRoute: config.cloud?.apiRoute,
      sitemap: config.sitemap,
      llms: config.llmsTxt,
      robots: config.robots,
    })
  ) {
    return method === "HEAD"
      ? HEAD({ url: event.url, request: event.request })
      : GET({ url: event.url, request: event.request });
  }

  return resolve(event);
}
```

## Custom route

If you want a custom MCP route, set it in `docs.config` and add the route file yourself.

```ts title="docs.config.ts"
export default defineDocs({
  entry: "docs",
  mcp: {
    enabled: true,
    route: "/api/internal/docs/mcp",
  },
});
```

<Callout type="warning" title="Custom routes are explicit">
  The package only auto-generates the **default** Next.js route at `/api/docs/mcp`. If you choose a
  custom route, keep that path in `docs.config` and update the framework public forwarder so the app
  and the MCP client point at the same endpoint.
</Callout>

Example custom Next.js route:

```ts title="app/api/internal/docs/mcp/route.ts"
import docsConfig from "@/docs.config";
import { createDocsMCPAPI } from "@farming-labs/next/api";

export const { GET, POST, DELETE, OPTIONS } = createDocsMCPAPI(docsConfig);

export const revalidate = false;
```

<Callout type="warning" title="@farming-labs/theme/api compatibility">
  `@farming-labs/theme/api` still works if you already rely on it, but prefer
  `@farming-labs/next/api` for Next.js routes going forward. The theme-level path will be
  deprecated. If `mcp.security` is configured, pass the imported live `docsConfig` to
  `createDocsMCPAPI`; the legacy no-argument form intentionally refuses to reconstruct executable
  authentication callbacks from source text.
</Callout>

When `search` is configured, the MCP `search_docs` tool uses that same adapter pipeline too. That
means your docs UI search and MCP search can share the same provider, chunking strategy, and custom
ranking logic instead of drifting apart.

That also means you can flip docs search itself over to MCP:

```ts title="docs.config.ts"
export default defineDocs({
  entry: "docs",
  search: {
    provider: "mcp",
    endpoint: "/mcp",
  },
  mcp: {
    enabled: true,
  },
});
```

For Ask AI only, keep the top-level search provider as-is and enable MCP retrieval in the AI config.
This uses the MCP server the docs site already exposes:

```ts title="docs.config.ts"
export default defineDocs({
  ai: {
    enabled: true,
    useMcp: true,
  },
});
```

If you changed the canonical MCP route, `useMcp: true` follows it automatically:

```ts title="docs.config.ts"
export default defineDocs({
  mcp: {
    route: "/custom/docs/mcp",
  },
  ai: {
    enabled: true,
    useMcp: true,
  },
});
```

For a hosted or external MCP endpoint, configure the endpoint on `ai.useMcp` instead. This does not
replace your LLM provider config; the MCP endpoint retrieves docs context, then Ask AI still sends
the final prompt to your configured model provider.

```ts title="docs.config.ts"
export default defineDocs({
  ai: {
    enabled: true,
    model: "gpt-4o-mini",
    apiKey: process.env.OPENAI_API_KEY,
    useMcp: {
      endpoint: "https://docs.example.com/mcp",
      headers: {
        Authorization: `Bearer ${process.env.DOCS_MCP_TOKEN}`,
      },
      toolName: "search_docs",
    },
  },
});
```

The custom MCP endpoint must support Streamable HTTP MCP and expose a search tool that returns docs
results. The default tool name is `search_docs`; override `toolName` only if your server uses a
different name. Stateful MCP servers may return an `mcp-session-id` on `initialize`; stateless
servers can omit it, and the docs client will only send the session header when one is provided.

For local self-hosted setups, relative MCP endpoints like `/mcp` and `/.well-known/mcp` are
supported. The canonical `/api/docs/mcp` route remains available too. The built-in `search_docs`
tool automatically falls back to simple search internally when it detects that same-route loop, so
the route stays usable for testing and local examples.

## Stdio transport

The same docs graph is available locally over stdio:

```bash title="terminal"
pnpx @farming-labs/docs mcp
```

Or with pnpm in an installed project:

```bash title="terminal"
pnpm exec docs mcp
```

By default the CLI reads `docs.config.ts[x]` from the project root. If your config lives somewhere else:

```bash title="terminal"
pnpm exec docs mcp --config src/lib/docs.config.ts
```

## What to ask it

Once your MCP client is connected, these are good first checks:

- `search_docs` for `feedback`, `page actions`, or `mcp`

## What the tools return

- `list_docs` returns page summaries grouped by section; pass `section` such as `getting-started` to narrow the result
- `list_pages` returns page titles, slugs, and URLs
- `get_navigation` returns the docs tree in a readable text outline
- `search_docs` ranks pages by title, description, and content matches
- `read_page` accepts either a slug like `installation` or a full docs path like `/docs/installation`; pass `section` for one heading and `maxChars` to cap both successful content and section-not-found output
- `get_code_examples` returns fenced code blocks with parsed metadata like `title`, `framework`, `packageManager`, and `runnable`
- `get_config_schema` returns `docs.config.ts` option metadata with paths, types, defaults, descriptions, and examples
- `get_context` deterministically selects matching sections, applies optional `framework`, `version`, and `locale` scopes, and reports source anchors plus conservative UTF-8 byte accounting

For `framework`, `version`, and `locale`, a page that declares a different value is excluded. A page
that omits a scope field is treated as general documentation and remains eligible for that scope.
Results are ranked deterministically, with source URLs resolving equal-score ties.

Declare scopes with optional top-level page frontmatter. Quote numeric versions so they remain
strings:

```md title="docs/guides/next-setup.mdx"
---
title: "Next.js setup"
framework: "nextjs"
version: "16"
locale: "en"
---
```

Every successful built-in tool result includes both MCP `structuredContent` validated against its
advertised `outputSchema` and the existing text content for clients that have not adopted structured
results yet.

Example tool calls:

```json title="list_docs"
{
  "name": "list_docs",
  "arguments": {
    "section": "getting-started"
  }
}
```

```json title="get_code_examples"
{
  "name": "get_code_examples",
  "arguments": {
    "query": "docs.config.ts",
    "framework": "nextjs"
  }
}
```

```json title="get_context"
{
  "name": "get_context",
  "arguments": {
    "query": "configure MCP search",
    "framework": "nextjs",
    "version": "16",
    "locale": "en",
    "tokenBudget": 4000
  }
}
```

`tokenBudget` defaults to `4000` and accepts values from `256` through `32000`. The dependency-free
budget strategy treats that value as a conservative UTF-8 byte ceiling: the complete assembled
`context`, including source headers and separators, never exceeds `tokenBudget` encoded bytes. The
response reports `strategy: "utf8-bytes"`, `maxUtf8Bytes`, `usedUtf8Bytes`,
`remainingUtf8Bytes`, and `conservativeTokenUpperBound`; each source reports its own `utf8Bytes`.
The upper bound intentionally overstates many tokenizer counts instead of presenting a
tokenizer-specific estimate as a hard guarantee.

Section parsing is shared by search, `read_page`, and `get_context`. It ignores headings inside
fenced code, supports ATX headings indented by up to three spaces and Setext headings, cleans inline
links and formatting from labels, and assigns stable `-1`, `-2`, and later suffixes to duplicate
anchors. Anchored search hits that no longer resolve are skipped instead of being attributed to
whole-page content. When a requested `read_page.section` is missing, `maxChars` also bounds the JSON
error and the included list of available headings.

```json title="get_config_schema"
{

    "option": "mcp.tools.listDocs"
  }
}
```

## When to use this vs markdown routes and `llms.txt`

- Use **markdown routes** when you want a plain HTTP URL for one page; in Next.js, `Signature-Agent` can return markdown from the canonical page URL too
- Use **`llms.txt`** when you want a static, crawler-friendly docs summary
- Use **`robots.txt`** when crawlers and AI agents need an explicit policy that allows docs and machine-readable routes
- Use **MCP** when you want a structured, queryable docs interface for agents and IDE tools

They complement each other well. Markdown routes are the simplest page-level HTTP surface,
`llms.txt` is lightweight and public, `robots.txt` makes access policy explicit, and MCP is
interactive and tool-based.
